# 翻訳タスク：構造化出力と非構造化出力

## 構造化出力 vs 非構造化出力の特性

構造化出力は求める結果を確実に取り出せる反面、フォーマット適合を強制することで
品質が低下する懸念がある。この問題意識から、翻訳実験では両方式を対比させている。

| 観点 | 構造化出力 | 非構造化出力 |
|------|-----------|-------------|
| 品質安定性 | 不要な出力が抑制され安定 | モデル・設定依存で不安定 |
| 暴走・途中切れ対応 | パース失敗で機械的にリトライ可 | 回答前の暴走でハルシネーション危険 |
| フォーマット制約の影響 | 一部モデルで品質崩壊（→ 非構造化が有効） | 一部モデルで暴走（→ 構造化が必須） |

**構造化出力における暴走・途中切れの処理**：

- **閉じ括弧が生成されない**：JSON パース失敗 → 機械的にリトライ可能。
- **閉じ括弧の後に空白が延々と続く**：JSON としては完結しているためリカバリー済み。
- **リトライの限界**：同じ入力・同じモデルで再試行すると同じ失敗が繰り返される可能性がある。
  温度パラメーターを変えるなどの工夫が必要なケースもある。

## 実験から得られた知見

翻訳実験（README.md）での構造化出力の影響はモデル特性に強く依存する：

- **構造化が有害**：gemma3-12b はレベル1（構造化推論）で23点に崩壊するが、
  同等内容を非構造化（tr6）にすると90点に回復（+67点）。ministral-3系・llama4-scout も同様。
- **構造化が不可欠**：qwen3-30b は非構造化出力で完全失敗（0点）。
  構造化出力がガードレールとして機能している。
- **高性能モデルは同等**：gpt-oss-120b・gemma3-27b は両方式で91〜92点と差がない。

直接翻訳（レベル0）ではモデルによって優位な方式が異なる（llama系・command-r-35b は非構造化
優位、qwen3-30b は構造化必須）。同等性能ならパース安定性から構造化を優先するのが基本方針。
推論付き翻訳（レベル1）では構造化制約が複雑な指示と組み合わさって崩壊するケースが目立つ。

## CoT・推論が翻訳品質を向上させない理由

コンテキストや語彙の検討が品質向上につながりそうに思えるが、翻訳実験では
推論を深めるほど改善するとは限らない。Qwen で `--no-think`（CoT 無効）の方が
安定するケースがあるのもこれと一致する。

考えられる理由：

1. **暗黙知の妨害**：高性能モデルは事前学習で文脈判断・語彙選択を暗黙知として習得済みである。
   CoT でそれを明示化させることは最適化された内部処理を「言語化」という別タスクに変換する
   ことになり、同じ結果が得られるとは限らない。

2. **中間誤りの固定化**：CoT で文法構造や話者の意図を誤って分析した場合、その誤りが
   コンテキストに固定されて最終翻訳に引き継がれる。直接翻訳では起きなかった誤りが
   推論を経ることで確定してしまう逆効果が生じる。

3. **注意の分散**：推論トークンが長くなるほど、最終翻訳を生成する時点での原文への注意が
   相対的に薄れる。モデルは原文よりも直前に生成した推論に引きずられやすくなる。

4. **選択肢の増加による不安定化**：語彙の候補を列挙して選ぶプロセスは、直接生成では
   生じなかった迷いを導入する。最適な翻訳は微妙な全体的バランスで決まることが多く、
   明示的な選択プロセスがかえって局所最適に陥る（README.md「複雑な推論の逆効果メカニズム」
   と一致）。

5. **タスクの性質**：ポッドキャストの短い発話はそもそも推論の余地が少なく、CoT が有効な
   「複雑な多段階推論」とは異なる。CoT は正答への道筋が明示的に存在するタスク（数学・論理）
   で効果を発揮するが、翻訳の語感・ニュアンスは言語化しにくい暗黙的判断に依存している。

すなわち自由記述先行（tr5/tr6）の価値は「品質向上」ではなく、**フォーマット制約による
崩壊を回避する手段**として位置づけるのが適切である。

## 非構造化出力を使う場合：2段構えアプローチ

構造化出力で品質が出ないモデルに対して、以下の2段構えが有効と考えられる：

1. **第1段階**：フォーマットを指示せずに自由出力させる（品質優先）
2. **第2段階**：自由出力から翻訳結果のみを取得（抽出専用）

translate5.py・translate6.py では構造化出力を使わずにテキストプロンプトで抽出を実装済み：

```
system: "Output only the requested translation text."
prompt: "Extract only the final <lang> translation from the following text..."
```

**抽出専用モデルの選定**：

同じモデルの再呼び出しに限らず、軽量モデルを固定採用することも選択肢である。
選定方法として、複数のモデルで同じ自由出力から抽出を実行し、多数決で一致した結果を
「正解抽出」とみなす機械的な方法が考えられる。評価に人手や正解データを必要とせず、
モデル間の合意を品質指標として利用できる点が利点。ただし翻訳結果が曖昧な場合は
多数決の「正解」自体が誤りになることもあり得るため、選定時には曖昧なケースを
意図的に含めた検証が必要である。

**実運用上の問題点**（tr5/tr6 のログより確認）：

- **第1段階で翻訳が生成されない**：推論のみで終わり末尾に翻訳が出力されない場合がある。
  抽出ステップで翻訳が存在しないと、モデルがハルシネーションで翻訳を捏造する危険がある。
  対策として抽出前に「回答が含まれるか否かの判定」を先行させる必要がある。
  構造化出力で `found: bool` のようなフィールドを先行出力させる設計も有効だが、
  判定ステップ自体も LLM に依存するため誤りが生じうる。
- **抽出時に語句が修正される**：「翻訳のみを取り出す」はずがモデルが独自に語句を
  書き換える例が散見される。第1段階の出力が忠実に反映されない。

**2段構えの適用限界**：

- tr5/tr6 自体が既に2段構えであり、抽出専用モデルの差し替えは改善幅が限定的な可能性がある。
- 構造化が必須なモデル（qwen3-30b など）では第1段階の自由出力自体が失敗するため成立しない。

## 実用上の結論

構造化出力は「単純だが堅牢」、非構造化＋抽出は「柔軟だが脆弱」という対比になる。
実用上は**構造化出力を基本とし、それで品質が出ないモデルにのみ非構造化（tr5/tr6）を採用する**
（README.md「モデル別実用設定一覧」）方針が合理的である。
tr5/tr6 は既に2段構えの実装であり、抽出専用モデルへの差し替えはさらなる改善の余地として
残るが、効果は限定的な可能性がある。

## 処理環境の前提

処理対象はデスクトップまたはクラウドであり、スマートフォンのような自己完結環境は対象外。
VRAM が十分にあれば MoE モデルが小型 dense モデルと同等の速度で動作するため、
敢えて小型モデルを選ぶ必要性は薄い。

## KV キャッシュの有効活用とサマリー圧縮の必然性

現在の翻訳実験（translate.py）はスライディングコンテキスト方式を採用しており、
毎回コンテキストの始点がスライドするため KV キャッシュが有効に機能しない。

一方、コンテキストを消去せずに全履歴を保持して積み上げる方式をテストしたところ、
履歴が40件付近に達した時点で処理が非実用的なほど遅延し、推論品質も低下した。
これは Transformer アーキテクチャの自己注意力機構（Self-Attention）の計算量が
コンテキスト長に対して2乗（O(N^2)）で増大するため、ローカルLLMの限られたVRAMと
計算リソースでは限界を超えてしまうことが原因である。

したがって、「全てを記憶させる」ことはローカル環境では物理的に不可能であり、
スライディング方式の「忘却（用語のブレ）」を解決するためには、古い履歴を高密度な
トークンに圧縮する**サマリー方式が構造的な必然**となる。

新アーキテクチャ（`~/llm/translate/translate-json.py`）では以下の設計でこの課題を解決し、
KV キャッシュを活用する：

- **システムプロンプトを固定分離**：会話履歴とは独立して常に先頭に置かれるため常時キャッシュ有効。
- **固定先頭によるサマリー圧縮**：スライディングではなく、一定件数に達したら古い履歴を
  「用語集＋要約」として圧縮し、`system + summary + 直近 N 件` という固定構造に再構築する。
  これにより総トークン数を安全圏に維持しつつ、圧縮後のコンテキスト始点が安定するため
  KV キャッシュが再利用される。
- **圧縮タイミングの最適化**：サマリー生成を圧縮直前（対象履歴がまだキャッシュに残っている間）
  に行うことで、圧縮後も直近 N 件のキャッシュが引き続き有効な状態を維持する。
