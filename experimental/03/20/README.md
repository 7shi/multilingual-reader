# threshold=20 試行

要約間隔 `--threshold 20`（デフォルト）、再編成まで `--keep 5` の設定で実施した試行です。

翻訳3回 × 評価3回（各 run の中央値）。

## KV キャッシュ動作の確認

実行ログの `prompt_eval_duration` で各モデルの KV キャッシュ効果を確認した。

| モデル | KV キャッシュ | 代表的な prefill tps |
|---|:---:|---|
| gemma4-26b | **有効** | 2行目〜: 992→1379→2174→3641 tps |
| gemma4-e4b | **有効** | 2行目〜: 3340→4592→6673→9131 tps |
| qwen3.6-27b | **無効** | 全行: 200〜235 tps（一定） |

gemma4 系は2行目から prefill tps が急上昇し、KV キャッシュが機能している。qwen3.6 は think/no_think の切り替えに内部タグ（`/think` など）を使う実装のため、Ollama が受け取るトークン列が毎回わずかに変化してプレフィックスが一致せず、キャッシュが無効になると考えられる。翻訳品質への影響はないが、処理時間が長くなる。

gemma4-26b は MoE（Mixture of Experts）モデルでアクティブパラメータが約 3B（A3B）のため、生成速度（eval tps）は gemma4-e4b と大差ない（いずれも 49〜51 tps）。一方で prefill tps は e4b の方が高い（全パラメータ数が小さいほど prefill コストも低いため）。KV キャッシュ効率・生成速度・翻訳品質の三点で、gemma4 系はハイブリッドモードとの相性が良い。

### 要約直後の cold start

要約生成後に履歴から要約を削除して翻訳を継続すると、直後の1行だけ cold start が発生する。Ollama の前のキャッシュは「要約込みシーケンス」なので、要約削除後のシーケンスとプレフィックスが不一致になるためである。

| モデル | 要約直後 cold start | 2行後のキャッシュ回復 |
|---|---|---|
| gemma4-e4b | 1.24s, 1501 tps | 0.07s, 30001 tps |
| gemma4-26b | （未測定） | （回復） |
| qwen3.6-27b | 2.06s, 912 tps | 0.23s, 8475 tps |

cold start は要約直後の **1行のみ** で、それ以降はキャッシュが回復する。

これは設計上の必要コストである。CoT による長い要約テキスト（1000+ トークン）を履歴に残すと、モデルが要約の文体・表現に引っ張られるリスクがある。履歴から削除することで要約の影響を後続の翻訳から分離し、翻訳スタイルを安定させる。1行の cold start はその代償として受け入れ可能。

### 主な問題と観察

**gemma4-26b-3: 100点**

5項目全て満点。experimental2 では「この評価者（qwen3.6）では97点が実質的な上限」とされていたが、今回はそれを超えた。3回の評価のうち2回が満点（eval-1, eval-2）、eval-3 も 97点と評価ブレが極めて小さい。`bachoter` の意訳など文化的ローカライズも高評価。

**gemma4-e4b-2: 68点急落**

主因は2つの術語誤訳：
- `grounding` → `puesta a tierra`（電気工学の「接地」。AI 用語としては `anclaje` や `vinculación contextual` が正しい）
- `bachoter` → `hacer trampas`（「ズル・インチキ」。正しくは「詰め込み勉強」）

これらが fluency・terminology・contextual_adaptation を一括して引き下げた。tr-1・tr-3 では同様の誤訳が出ておらず、glossary の初期蓄積のブレによる一時的な崩壊と見られる。

**qwen3.6-27b-3: 87点急落**

主因は `prompt` → `indicio`（スペイン語で「手がかり・痕跡」）。AI コンテキストでは `prompt` のままか `instrucción`/`entrada` が正しく、3回の評価全てで指摘された。tr-1・tr-2 では出ていないため、run-3 の glossary 初期蓄積で誤訳が固定・強化されたと考えられる。

glossary 方式は用語の一貫性を高める一方、**初期の誤訳が蓄積・強化されるリスク**があることをこの2ケースが示している。

### 総評

- **gemma4-26b** が最も安定。3 run の中央値は 96・96・100 で翻訳ブレが最小。参照訳（97点）を上回る run が存在し、実用レベルとして最高評価。
- **qwen3.6-27b** は KV キャッシュが効かず処理時間が長い上に、tr-3 で術語誤訳による急落があり安定性に懸念が残る。
- **gemma4-e4b** は tr-1・tr-3 で 94〜96 点と優秀だが、tr-2 の急落（68点）で信頼性に疑問符。glossary の初期蓄積次第で結果が大きく左右される。
- ハイブリッドモード（CoT 要約）が翻訳品質に与えた正の効果は gemma4-26b の 100 点が示す一方、CoT を持たない要約は初期誤訳を固定するリスクも持ち合わせており、今後の課題として残る。
