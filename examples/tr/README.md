## モデル選択と検証の背景

- **追加検証の目的**: gemma4 の結果において、一部の言語で異様に低い得点が確認されました。これを踏まえ、翻訳モデルを gpt-oss や qwen3.6 に変更し、翻訳品質が改善するかを検証するためのセットを設けています。
- **評価モデルの統一**: 品質を相対比較するためには、採点基準（定規）を全セットで統一することが最も重要です。そのため、すべてのセットにおいて評価モデルを qwen3.6 に統一しています（自己評価を含む）。

この結果から、言語の性質（語族や学習リソースの量）によって最適なモデルが明確に異なることが分かります。

**高品質な翻訳（90点以上）を達成した言語とモデルの傾向**:

- **国際的なメジャー言語と欧州主要言語の安定性**: アラビア語（96点）は、非連結的な形態論という複雑な構造を持ちますが、国連公用語でもある国際的なメジャー言語として極めて豊富な学習リソースが存在するため、gemma4 による高品質な出力はリソース量に見合った妥当な結果と言えます。フランス語、スペイン語、イタリア語、ポルトガル語、ロシア語に加え、ベトナム語も少なくとも 2 モデルで 90 点台を維持しました。今回追加した北欧言語でも、デンマーク語は gemma4 で 92 点に達しており、学習資源が比較的厚い欧州言語群では依然として高い安定性が見られます。詳細は [MEMO.md](../../MEMO.md) を参照してください。
- **低〜中リソース言語における gemma4 の安定性**: 朝鮮語、フィンランド語、セルビア語、トルコ語に加え、ウクライナ語やポーランド語でも、gemma4 は他の巨大モデルを抑えて圧倒的な高得点を記録しました。今回追加したクロアチア語でも 89 点、マレー語でも 86 点と優勢で、ベトナム語では 97 点と極めて高く、言語構造が複雑であっても文法の骨格を保ち、ポッドキャストの自然なトーンを維持する能力に長けています。
- **文字体系と語族の影響**: アラビア文字を使用する右横書き（RTL）言語において、アラビア語（96点）やペルシア語（96点）では高品質な翻訳が可能でしたが、ウルドゥー語（72点）では語彙ミスなどにより精度が低下しました。また、セム語族のヘブライ語（89点）でもアラビア語ほどの安定性は見られませんでした。特殊な文字体系や語族の構造自体が障害になるというよりも、モデルが事前学習で取り込んだリソースの絶対量が翻訳品質に直結していることが示唆されます。

**スコアが低下・崩壊したモデルの各論（ハルシネーションと特異点）**:

- **gpt-oss と qwen3.6 の幻覚の深刻さ**: パラメータ規模が大きく一部の言語では優秀な両モデルですが、リソースの少ない言語になると、翻訳テキスト内に「他言語（朝鮮語やアラビア語など）の混入」「システムプロンプトの漏洩」「漢字の混入」といった致命的な構造破壊を頻繁に引き起こし、スコアが暴落しました。アラビア語における qwen3.6 の「姆（ム）」という漢字の大量混入はその典型例です。また、日本語においても qwen3.6 で中国語の単語が不自然に混入し文章が破綻する事象が確認されています。新たに確認したマラヤーラム語でも、qwen3.6 は正書法の崩れと壊れた語形が連続し、読み物として成立しないレベルまで劣化しました。
- **インターリングアのモデル間ギャップ**: インターリングアでは、gpt-oss だけが 73 点で可読域を維持した一方、gemma4（20点）と qwen3.6（32点）はポルトガル語・スペイン語・英語などの断片を混ぜたコードスイッチングを起こして崩壊しました。学習リソースが薄い人工言語では、モデル間の事前学習差が極端に出ることを示しています。
- **gemma4 におけるハンガリー語の特異な崩壊**: 他の低リソース言語で抜群の安定性を見せた gemma4 ですが、ハンガリー語においてのみ致命的なコードスイッチング（幻覚）を起こし自滅しました。文法構造は維持しているものの、文中に突然、英語（absolutely）、ギリシャ語（πρέπει）、シンハラ語（ ின்）、ベトナム語（ấy）、ロシア語（себе）などの全く関係のない言語の単語がランダムに混入する奇妙な現象が確認されています。これはハンガリー語の学習データのベクトル配置が、このモデルの潜在空間においてのみ他言語と極端に混線しやすい状態になっていることを示唆しています。
- **構造的難易度とリソース不足の壁（タミル語など）**: タミル語、カンナダ語、エストニア語、マラヤーラム語などは、全モデルを通して実用レベルに達しませんでした。特にタミル語は固有の文字体系と複雑な膠着語の性質を持ち、学習リソースの不足と相まって、ベースラインである gemma4 でも英語の混入や綴りミスが多発し、出力が破綻しています。今回追加したタイ語も 3 モデルすべてが 50 点未満に沈み、内容保持はできても正書法の崩れで可読性が失われました。アイスランド語でもラテン文字圏でありながら全モデルで文法崩れと直訳調が強く、学習資源の薄さが品質に直結していることが見て取れます。ラトビア語とリトアニア語でも、gemma4 が多言語混入や `<|image>` トークン漏出を起こして大きく崩れ、バルト諸語への適応の弱さが表面化しました。

万能な翻訳モデルは存在せず、言語に合わせてモデルを使い分けることの重要性が強く示唆されています。

## 評価設計の方針：話者タグを分離しない理由

本検証では、対話スクリプトの話者タグ（`Camille:` や `Luc:`）を翻訳対象から切り離して後付けすることは、現在は行っていません。

過去には、翻訳対象を本文だけに限定し、話者タグはロジック側で後付けする設計も試していました。この方式であれば、タグ脱落による減点を避けやすく、純粋な本文の訳質だけを測ることができます。

しかし、その場合は「モデルが長い対話スクリプトの構造を最後まで維持できるか」という、実運用で重要な能力を評価から外してしまいます。話者タグの脱落は単なる表面的な整形ミスではなく、長い出力の中で構造を保持し続ける注意力や安定性の不足として現れることがあるためです。

また、話者タグだけを例外的に保護すると、通常の本文で起こり得る行脱落・文脱落・対応関係の崩れとの境界が曖昧になります。実際の運用では、タグだけでなく本文そのものにも欠落や取り違えが起こり得るため、タグだけを別扱いするとモデルの信頼性を過大評価するおそれがあります。

そのため本READMEの比較では、「意味の正確さ」だけでなく「スクリプト全体を壊さず出力できるか」まで含めて翻訳品質として扱っています。特に `gpt-oss` のように内容面では良好でも話者タグの保持で減点されるケースはありますが、それも含めて実用上の耐性差として記録しています。

## 観察と考察：LLMの読み書き能力の非対称性

本検証過程で、LLMの「評価能力（読む力）」と「翻訳能力（書く力）」の間に著しい非対称性が観察されました。この現象は、人間の外国語学習における受容能力と産出能力のギャップ（読んで理解できても、正しく書くのは難しい）に似ています。

**qwen3.6の事例**:
- 翻訳の評価では、文法エラーや不自然な表現を的確に指摘し、理路整然とした妥当な評価を下しました。
- 翻訳の生成では、ヒンディー語などで文法が破綻しスペルミスが多発する低品質な出力となりました（書く力の未成熟）。

### 自己評価 vs 他者評価の比較検証

当初、qwen3.6 が生成した翻訳の評価では、自己評価のバイアスを避けるために別モデル（gpt-oss）を評価者として試行していましたが、その結果から以下のことが判明しました。

1. **自己評価バイアスの不在**: qwen3.6 は、自分自身が生成した破綻したテキストを評価（自己評価）させた際にも、甘く採点することなく、客観的で極めて厳しい評価を下しました。
2. **採点基準の違い**: 両モデルとも「プロンプトの混入や誤訳などのエラーを正確に検知・指摘する能力（読む力）」については同等に高く機能していました。しかし、そのエラーに対する**採点基準（Scoring Criteria）**に極端な違いがありました。gpt-oss は意味が通じれば寛容に高得点を与えるのに対し、qwen3.6 は文法的な破綻を容赦なく減点しました。

これらの結果から、LLMの自己評価においては必ずしも強いバイアスがかかるわけではなく、読み書き能力の非対称性が如実に表れることが分かりました。そして、モデルごとに「エラーに対する採点の甘さ」が根本的に異なるため、品質の相対比較を行う上では「自己評価バイアスの回避」よりも**「同一の評価モデル（定規）を用いて採点基準を統一すること」の方が遥かに重要**であるという強力な裏付けとなりました。

### 実行環境（プロバイダー）の違いによる挙動の差異

[gpt-oss/](onde/gpt-oss/) の検証では、同一モデル（qwen3.6, gpt-oss）をローカル（Ollama）とクラウドAPI（OpenRouter）の独立したパイプラインで比較しました。その結果、以下の運用上の重要な知見が得られました。

1. **エラー検知能力は環境に依存しない**: プロバイダーや量子化の有無が異なっていても、評価モデルがテキストからバグ（文字化けやプロンプト漏洩など）を検知・指摘する能力は完全に一致しました。
2. **翻訳生成のハルシネーションは環境に依存しない**: マイナー言語の翻訳において gpt-oss が引き起こす「他言語の混入」や「システムプロンプトの漏洩」といった致命的なバグは、ローカルでもクラウドでも共通して発生しました。モデル固有の能力の限界（書く力の未成熟）が原因です。
3. **構造化出力（JSON）のバグの出方は環境に依存する**: クラウド（OpenRouter）ではJSONの特定フィールドに無意味な値が入り出力が破損するケースが目立った一方、ローカル（Ollama）では無限ループに陥るケースが目立ちました（こちらはツール側の自動リトライで救済可能）。

完全な自動化パイプラインを構築する上では、モデルだけでなく**稼働環境（プロバイダー）まで含めて「定規」を固定すること**、そしてプロバイダーごとの「ハルシネーションの出方のクセ」を考慮したエラーハンドリングが必要不可欠です。
